Ruby Hashes and JavaScript Objects: Secret Twins

February 7, 2015

This week, we've been expanding our knowledge of object-oriented languages by diving into JavaScript. After spending 4 weeks on Ruby, learning the syntax and getting comfortable with object-oriented programming, JavaScript has been that much easier to learn, but it hasn't been without its pitfalls. As both languages are object-oriented, they share several core similarities that makes learning one easier when you're familiar with the other and the object oriented principles underlying both languagues guide design choices in similar ways, but they also differ in some pretty fundamental ways that can throw you for a loop if you're not prepared for it. There are too many to list in one post, but one in particular that I found very interesting was the difference in Ruby hashes vs. JavaScript hashes...which are actually nonexistent.

Find that shocking? So did I.

Ruby Hashes

Ruby hashes are straightforward, easy to define, easy to manipulate, and come with a host of helpful built-in methods, as every hash in Ruby is an instance of the Hash class. If you're unfamiliar with hashes at all or just need a refresher: a hash is a dictionary-like collection of unique keys and their paired values. Implicit creation in Ruby looks like this:

Using the built-in methods you can then do things with a hash like iterate through it and perform an operation on either all of the keys, all of the values, all of the pairs of key and values, or only some of any of those. Really, whatever your particular needs are. In addition, because they are objects, they are conferred with all of the powers that objects in Ruby possess.

JavaScript...hashes, except they are not a thing

And then we have hashes in JavaScript which do not exist under that label. Unlike Ruby, JavaScript has no dedicated hash class from which to create new hash objects. Instead, it just has objects, which look and in most ways and on most levels, act, just like hashes. For most intents and purposes, JavaScript objects are hashes, but they are not called such and have a bit more going on.

According to the book Eloquent JS: "Inside the curly braces, we can give a list of properties separated by commas. Each property is written as a name, followed by a colon, followed by an expression that provides a value for the property. Spaces and line breaks are not significant. When an object spans multiple lines, indenting it like in the previous example improves readability. Properties whose names are not valid variable names or valid numbers have to be quoted."

If it looks like a hash, and smells like a hash...

These objects then behave just like objects in any object oriented programming language: they hold state and can receive messages. The point of contention surrounding objects in JavaScript is a perceived conflict of naming. When, for most intents and purposes, JavaScript objects are behaving just like hashes and are hashtables internally, why aren't they just called hashes? It's a matter that's up for constant debate and further digging can probably get you closer to some solid conclusions.

If you're like me and longing for the comforting familiarity of Ruby hashes in JavaScript, you can simply define your hash as an object and then set your keys as properties.

Learning JavaScript has definitely been frustrating and counterintuitive at times compared to the ease with which I took to Ruby, because major differences like the one we've explored with hashes can be jarring and confusing and make it difficult to grasp the concepts, but the jury is still out on how I feel about JavaScript as a whole. Time will tell.
