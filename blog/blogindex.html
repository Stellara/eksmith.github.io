An Intro to Ruby Classes

January 25, 2015

The world is filled with objects that interact with each other and many of these objects are produced from some template that makes them the same in many ways, but unique in others. To make a cookie, for example, you follow some recipe that is generally the same for all cookies, but you tweak the amounts of butter, sugar, or chocolate to get different kinds of cookies. The existence of objects that have attributes (like amount of butter or sugar) and that have behaviors that they can do and that can be done to them (like baking or being eaten) is at the core of object oriented programming.

In Ruby, we can model real world objects by using classes. In Ruby, "Cookie" would be a class and "chocolate chip cookie" would be an instance of it. A class is the general template for creating objects of its kind. An instance of that class is the specific object. Let's look at this in code:

To get started with a class, all you need is the class keyword, a class name that follows the naming convention by beginning with a capital letter, and the end keyword. This is just a skeleton. The working parts of your class will go in between these two lines.

Initializing a Class

It's not always necessary, depending upon what you're trying to achieve, but you may want to initialize your class.

Initialize is a special method in Ruby programs. When you call Song.new to create a new Song object, Ruby creates an uninitialized object and then calls that object's initialize method, passing in any parameters that were passed to new. This gives you a chance to write code that sets up your object's state. - Pragmatic Programmer's Guide to Ruby

This means that you will be setting variables that represent characteristics that you want each instance to have their own copies of upon creation. These characteristics are known as instance variables. Each time that you create a new instance of the class, you will give it its own specific values for these variables:

In the above example, we initialize Cookie with an amount of sugar, an amount of butter, and an amount of flour. We set chocolate to 0 by default because while every instance of cookie will have sugar, butter, and flour, not every instance will have chocolate. Different kinds of cookies have different ratios of these ingredients, but they share the fact that they are made with these ingredients.

Instance Variables

When we create the chocolate_chip_cookie instance of cookie, we pass in the amounts of ingredients that are unique to chocolate chip cookies. This is what instance variables track - data about an instance of a class that is unique to that instance. Additionally, we will do this each time we create a new instance of Cookie.

Instance variables follow the variable naming convention of being lowercase, but begin with the @ symbol. Scope is extremely important when considering whether or not your class should have instance variables and what they should be. This simply means that we must consider where in the program the variable is accessible. Instance variables are only accessible to the object that they belong to. This means that if chocolate_chip_cookie changes its amount of sugar, it is the only instance that will be affected by that change. If we had a sugar_cookie instance of Cookie, its amount of @sugar would be untouched.

Instance Methods

When we call it on chocolate_chip_cookie, we get false, because it doesn't have enough sugar and butter to be yummy. When we call it on the sugar_cookie instance of the Cookie class, since sugar_cookie has its own state independent of chocolate_chip_cookie's, we get a true result. Looks like we need to tweak our recipe for chocolate chip cookies!

When would you want to use a class and why?

The Well-Grounded Rubyist has wonderful advice on using classes:

Definining a class lets you group behaviors (methods) into convenient bundles, so that you can quickly create many objects that behave essentially the same way.1

A class is a factory for creating multiples of an object and allows for more complex interactions between objects. If you don't need to have several instances of an object, then you probably do not need a class. The class itself is useful when you're creating objects from it.
