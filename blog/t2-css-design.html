Positioning an element using CSS is like bringing a tribble back to the Enterprise. It won't stay where you put it and before you know it you have more elements in places they were never supposed to be.It seemed like a cute and cuddly idea at the time. (I'm sure there's a better analogy, and any other Trekkies out there feel free to make one, but it won't be as cute.) The problem with using positioning is that if you attempt to use it without very clearly understanding what each value does, you might make more problems for yourself and make your page look worse than it would have without any positioning at all. You can generally strong arm elements where you want them to be through trial and error, by applying different values to a selector until it looks how you want it to, but that can be tedious, makes fixing bugs difficult, and can have a domino effect later if you change some other related element's position. Allow me to pilot you through this murky quadrant of web development space and we might just make it back to Earth. A preliminary scan of "space" To begin with, let's get the lay of the universe we're working in. Elements are presented on a page according to their normal flow. This is just where they fall into place without positioning, sort of like the effect of gravity on objects in the physical world. The position property in CSS allows you to move elements where you want them to be visually, and to put them out of flow if you wish. The name is pretty clear and that's what you would expect for it to do. Going into the specific values that this property can take is where it gets complicated. Those values for our purposes are static, relative, absolute, and fixed. Static Anything that is not positioned is static. Elements have the static value by default if you don't give them any other value. This means they will just stack together in the normal flow that you have laid for them in your html. HTML
